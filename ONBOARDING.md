# ONBOARDING ‚Äî embedded-zhc

You are an AI agent arriving on **embedded-zhc**, the static-memory
C++20 Zigbee device library that powers ZHAC's device coverage. Read
this top-to-bottom before making changes.

---

## 1. Platform context (ZHAC in one page)

**ZHAC** = dual-chip ESP32 Zigbee Home Automation Controller.

- **ESP32-P4** runs Zigbee coordination (firmware: `zhac-main-core`).
- **ESP32-S3** runs WiFi / REST / WS / MQTT (firmware: `zhac-net-core`).
- They talk over SPI via a custom binary framing called **HAP**
  (not HomeKit ‚Äî our own protocol).
- A Preact SPA (`www-spa`) is bundled into S3 SPIFFS and speaks WS.

`embedded-zhc` is the **device knowledge base**: it tells the P4 how
to recognise a newly-joined Zigbee device, what attributes it
exposes, and how to encode/decode its ZCL traffic.

The 7-repo split (as of 2026-04-23, tag `v2026042301`):

| Repo | Role |
|------|------|
| `zhac-platform` | Meta (4 submodules + umbrella LICENSE/CLA) |
| **`embedded-zhc`** *(this)* | Apache-2.0 ZCL device library |
| `zhac-components` | Shared ESP-IDF components (incl. `zhc_adapter`) |
| `zhac-main-core` | P4 firmware (AGPL) |
| `zhac-net-core` | S3 firmware (AGPL) |
| `www-spa` | Preact Web UI (AGPL) |
| `zhac-docs` | Public docs |

## 2. What this library gives you

- **`zhc::PreparedDefinition`** ‚Äî static-memory device record.
  Everything needed to match + drive a Zigbee device.
  - `zigbee_models[]` + `manufacturer_names[]` / `manufacturer_name_prefix`
    ‚Üí fingerprint matching
  - `exposes[]` ‚Üí surface the UI binds to
  - `from_zigbee[]` / `to_zigbee[]` ‚Üí ZCL ‚Üî `ZclAttribute` converters
  - `bindings[]` ‚Üí clusters to `bind` during commissioning
  - `reports[]` ‚Üí `configureReporting` specs
  - `config_steps[]` ‚Üí initial reads after bind

- **`zhc::DefinitionRegistry`** ‚Äî O(1) lookup by ZCL model +
  manufacturer. Two-pass: Pass 1 matches definitions with
  manufacturer filter, Pass 2 falls back to model-only matches.

- **`zhc::FzConverter` / `zhc::TzConverter`** ‚Äî decode / encode
  pipeline for ZCL attributes. Generic converters
  (`kFzOnOff`, `kFzBrightness`, `kFzColorTemperature`, `kFzColor`,
  `kFzPowerOnBehavior`, `kTzOnOff`, `kTzBrightness`, `kTzColorTemp`,
  `kTzPob1`, `kTzLock`, `kTzCoverState`, `kTzCoverPosition`, ‚Ä¶) live
  under `definitions/_generic/`.

- **`zhc::Expose`** ‚Äî the attribute-surface record, with
  `ExposeType` (Binary, Numeric, Enum, List), `Access` bitmask
  (bit0=STATE, bit1=SET, bit2=GET), optional unit/enum-values, and a
  trailing `ExposeCategory` (`State` / `Config` / `Diagnostic`) that
  the UI uses to route between States and Options tabs.

- **Exposes-bindings JSON generator** ‚Äî builds JSON descriptions of a
  device's controllable surface. Consumed by `zhc_adapter_build_exposes_json`.

- **Zero dynamic allocation, no exceptions, no RTTI.** Fits on MCU.

### Important: `generated/` is committed

Files under `definitions/**/generated/` are **regenerated from the
zigbee-herdsman-converters source tree** by tools in the private
`zhac-tools` repo. Run `just refresh-parity` to refresh the whole
library (requires access to `zhac-tools`). Manual edits to
`generated/` are routinely overwritten ‚Äî if a device needs hand-tuning,
either extend the hand-written sibling file or move the device to a
non-generated location.

## 4. Building

### Standalone (host tests)

```bash
cd embedded-zhc
cmake -S . -B build -DZHC_BUILD_TESTS=ON
cmake --build build -j
./build/tests/zhc_tests
```

Warnings-as-errors are enforced (`-Werror`). Library is currently
header-heavy; `CMakeLists.txt` will switch to `STATIC` once more code
moves out of headers.

### As an ESP-IDF component (from firmware)

Firmware repos pull `embedded-zhc` via CMake `FetchContent` at a
pinned tag, overridable with `EMBEDDED_ZHC_PATH`:

```bash
cd zhac-main-core
EMBEDDED_ZHC_PATH=$HOME/webapp/zhac/embedded-zhc idf.py build
```

`embedded-zhc` is **not** an ESP Component Manager component ‚Äî it
ships its own CMake tree and is pulled in at the root-CMake level by
both firmware repos.

## 5. The device-definition recipe

A typical definition file (e.g.
`definitions/ysrsai/generated/Ysr_YSR_MINI_01_rgbcct.cpp`) wires up:

1. `zigbee_models[]` ‚Äî ZCL Basic cluster `modelId` strings.
2. `manufacturer_names[]` or `manufacturer_name_prefix` ‚Äî optional
   manufacturer filter (required when two vendors share a `modelId`).
3. `exposes[]` ‚Äî UI surface, with `ExposeType`, `Access`,
   `ExposeCategory`.
4. `from_zigbee[]` ‚Äî pointers to `FzConverter` (typically generic
   ones under `_generic/`).
5. `to_zigbee[]` ‚Äî pointers to `TzConverter` for each writable
   expose.
6. `bindings[]` ‚Äî clusters to bind after pairing.
7. `reports[]` ‚Äî `configureReporting` specs so the device emits
   change reports (critical for Tuya LED drivers that stay silent
   otherwise).
8. `config_steps[]` ‚Äî initial reads after bind, so the States tab
   isn't empty until the user physically toggles the device.

Then the definition is registered via an `extern const
PreparedDefinition kDef_*` instance ‚Äî the registry walks these at
startup.

## 6. Conventions

- **C++20**, zero allocation, no exceptions, no RTTI, no TLS.
- **All arrays are `constexpr`** ‚Äî definitions live in flash, not
  RAM.
- **`static_assert` on struct sizes** ‚Äî `ZclAttribute` must stay at
  52 bytes; bumping it requires bumping `NVS_SHADOW_VERSION` in
  `device_shadow`.
- **String keys ‚â§ 20 chars, string values ‚â§ 24 chars.** If you need
  more, either shorten the expose name or widen the union (requires
  schema bump everywhere).
- **Use generic converters whenever possible.** If the device is a
  stock Zigbee light, `kFzOnOff + kFzBrightness + kFzColorTemperature
  + kFzColor + kFzPowerOnBehavior` + their TZ counterparts covers
  ~95 % of lights. Custom converters are a last resort.
- **Match z2m behaviour** when in doubt ‚Äî check the cited
  `z2m-source:` comment in each definition and cross-check against
  `zigbee-herdsman-converters` upstream.

### kFzBattery scaling (memorise)

- `voltage` = `v->u * 100` (raw attr is in units of 100 mV ‚Üí store mV)
- `battery` = `v->u / 2` (raw attr is half-percent ‚Üí store %)

This trips up almost every new converter.

## 8. Gotchas

- **Device coverage expectation**: 98.9 % full parity with
  zigbee-herdsman-converters (4 167 / 4 213 devices). If you add a
  new vendor, run `just refresh-parity` in the monorepo clone (or
  private `zhac-tools`) to regenerate the bindings-JSON and parity
  stats.
- **Fingerprint collisions**: `ZB-CL01` is shared by Kurvia, Lonsonho,
  and Ysrsai. The manufacturer-filtered variants sit in Pass 1; the
  naked `Kurvia` falls through in Pass 2. Order matters.
- **TZ type union**: when implementing a new TZ converter, accept
  `Bool`, `Uint`, `Int`, and `StringRef` ‚Äî UI paths send `Uint` via
  `zhac_adapter_send_uint`, older converters that only accepted
  `Bool`/`StringRef` silently rejected valid input.
- **Expose categorisation**: set `ExposeCategory::Config` for things
  like `power_on_behavior` that belong on the Options tab, not the
  States tab. `State` is default. `Diagnostic` for read-only
  housekeeping (link quality, battery).
- **`configureReporting` + `config_steps` are mandatory for lights.**
  Without them, Tuya-style LED drivers never send attribute reports
  and the UI stays stuck on the last pre-toggle value.

## 9. User preferences (persistent)

- **User builds firmware themselves.** Don't run `idf.py build`.
- **Early-dev stance.** Breaking ABI changes are fine; bump
  `NVS_SHADOW_VERSION` and ship.
- **Prefer hook registration** when a library consumer needs
  behaviour injection ‚Äî see `zhac_adapter_register_shadow_hook` in
  `zhac-components` for the canonical pattern.
